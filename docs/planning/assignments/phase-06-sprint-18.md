## IV. Sprint Objective

Implement contract-backed OIDC client metadata lifecycle management.

Sprint 18 establishes persistent OIDC client metadata under `src/modules/oidc`, with admin-facing orchestration through approved service contracts.

Sprint 18 must ensure:

* OIDC owns client metadata lifecycle.
* Admin orchestrates client management but does not own client persistence.
* Client secrets are generated by the system.
* Client secrets are stored hash-only.
* Raw client secrets are returned only once during create or rotate.
* Redirect URI validation remains exact-match.
* Managed DB clients integrate safely with existing OIDC client validation.
* Existing static config client fallback may remain to avoid breaking existing flows.
* Client lifecycle mutations emit audit events without secret leakage.

### Task 93 - OIDC Client Service

Goal:

* implement OIDC-owned client lifecycle business logic.

Required service operations:

* create client
* get client
* list clients
* update controlled client metadata
* disable client
* rotate client secret
* validate managed client for OIDC flow

Required rules:

* generate `client_id` inside service or approved utility
* generate client secret inside service or approved crypto utility
* hash client secret through approved infrastructure crypto/hash abstraction
* return raw client secret only once on create
* return raw client secret only once on rotate
* never expose `client_secret_hash` in safe client view
* validate redirect URIs
* validate allowed scopes
* validate allowed grant types
* validate response types
* reject forbidden update fields
* reject disabled clients in validation path
* preserve static config fallback if required
* avoid token/session lifecycle mutation

Acceptance criteria:

* client service owns all client lifecycle rules
* create returns raw secret once and persists only hash
* rotate returns raw secret once and persists only hash
* get/list/update never return raw secret or secret hash
* disabled clients fail managed-client validation
* redirect URI validation is exact-match
* grant type / response type / scope validation is deterministic

### Task 95 - Managed Client Validation Path

Goal:

* integrate DB-managed clients into the existing OIDC client validation path without breaking current static config clients.

Required behavior:

* check DB-managed client by `client_id`
* if managed client exists:

  * require status `active`
  * validate redirect URI by exact match
  * validate client secret through hash comparison when credential validation is required
  * validate allowed scopes/grant types/response types as required by flow
* if managed client does not exist:

  * use static config fallback if retained
* static config fallback must not override a disabled managed client with the same `client_id`
* invalid client must be rejected deterministically

Rules:

* managed client lookup remains inside `oidc`
* no direct user DB access
* no token/session behavior change beyond approved client validation integration
* fallback behavior must be documented in report

Acceptance criteria:

* existing static client flow remains working if fallback is retained
* managed active client validates successfully
* managed disabled client is rejected
* unknown client is rejected
* redirect URI mismatch is rejected
* wildcard/prefix/substring redirect matching is not introduced

* `oidc.client.created`
* `oidc.client.updated`

Implementation rule:

* at minimum, each client mutation must record one safe audit event
* if using only admin-facing events, document rationale

Audit payload rules:

* metadata is bounded

Acceptance criteria:

* client create emits safe audit event
* client update emits safe audit event
* client disable emits safe audit event
* client secret rotate emits safe audit event
* audit events contain no raw secret or secret hash
* audit failure behavior follows audit service contract
